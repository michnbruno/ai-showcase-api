March Deep Dive Plan

v3.0 Active Updated: Mar 10, 2026  ·  Target: End of March 2026
North Star

Build a single deployable Azure AI Showcase Application — one GitHub repo, one resource group, one Bicep deployment — demonstrating how Azure AI services map to real enterprise business problems across 6–7 primitives. Learn by doing. Each part builds on the last. End state: a working app, a portfolio entry, and every line of the AI Competencies resume section backed by real hands-on experience.

The Four Parts
Part 1

Document Pipeline

User uploads a document. That single action fans out to three primitives automatically — extraction, understanding, retrieval. This is the core enterprise AI pattern and the highest ROI starting point.
Document Intelligence AI Language AI Search / RAG
Build Targets
  • 1a — Document Intelligence extraction — structured fields out ✓
  • 1b — Language Service — entities, sentiment, key phrases
  • 1c — AI Search indexing — document made retrievable
  • 1d — Upload UI — drag/drop PDF or form
  • 1e — .NET service layer — one service class per primitive
  • 1f — Results panel — show extracted + analyzed output
Primitives Covered
  • #1 Document Understanding
  • #2 Text Understanding
  • #3 Knowledge Retrieval
  • Architecture pattern: event-fan-out from single upload trigger
  • Before/after comparison — what this replaced in custom code
Part 2

Interactive & Media

A chat/media panel demonstrating synchronous AI interactions. Speech input, vision analysis on dropped images, and OpenAI conversational — three primitives surfaced through one interactive UI surface.
Azure OpenAI Azure Speech Azure AI Vision
Build Targets
  • 2a — Chat panel — OpenAI .NET SDK integration
  • 2b — Voice input — Speech-to-text feeding the chat
  • 2c — Image drop — Vision analysis with labeled results
  • 2d — Shared UI panel, tabbed or sectioned layout
  • 2e — OpenAI namespace / NuGet — validate the .NET story
Primitives Covered
  • #4 Conversational AI (necessary coverage, not focus)
  • #5 Speech Processing
  • #6 Vision Perception
  • Architecture pattern: synchronous request/response
  • OpenAI as cross-cutting service — note it touches 4 primitives
Part 3

Platform & Safety Dashboard

The broader primitives that operate at platform level — moderation, anomaly awareness, safety guardrails. These are results/status surfaces, not interactive flows. A dashboard panel is the right pattern here.
Content Safety Azure ML (awareness) Foundry Agents (survey)
Build Targets
  • 3a — Content Safety — run outputs from Parts 1 & 2 through moderation
  • 3b — Results dashboard — safety scores, flags, classification
  • 3c — Foundry Agents orientation — survey level, understand what's there
  • 3d — Azure ML — note capabilities, document without deep dive
Primitives Covered
  • #7 Forecasting (awareness only)
  • #8 Anomaly Detection (awareness only)
  • #11 Planning / Reasoning — Foundry Agents survey
  • #12 Safety / Moderation — hands-on
  • Note: #7, #8, #9 are Azure ML territory — flag, don't deep dive

Synthesize, document, publish. The app is built — now make it presentable and make the learning visible. This part produces the artifacts that make the month professionally reusable.

App Insights
Guardrails
  • Set Azure Cost Management budget alert at $75 before any deployments
  • Use consumption/free tiers during build — tear down when not in use
  • Job search stays background — 1–2 hrs every couple days
  • Azure ML is awareness only — it's a separate discipline, don't get pulled in
  • Chatbot / conversational coverage is necessary but not the star of the show
  • Agents / Foundry — survey level only, it's moving too fast to go deep
  • If a service pulls you deeper than planned, follow the energy — plan pivots
  • Parts are sequential but not rigid — revisit earlier parts as learning compounds
Version & Pivot Log
v1.0
Mar 8
Initial plan — week-based hybrid structure. 12 primitives framework adopted as curriculum backbone. Four calendar weeks: broad survey first two, deep dive last two. Primary deliverable: Azure AI Primitives Assessment document.
v2.0
Mar 8
Pivot to build-first, parts-based structure. Calendar weeks replaced with four logical parts. Primary deliverable changed from assessment document to working Azure AI Showcase Application — one repo, one resource group, 6–7 primitives composed into three UI scenarios. Assessment becomes the write-up that accompanies the app. Rationale: learn by doing better matches existing Azure depth; single unified project avoids scattered sample repos; app is stronger portfolio artifact than document alone.
v3.0
Mar 10
Build environment pivot to Visual Studio + numbered build targets. Replaced VS Code / CLI approach with full Visual Studio for initial build phase. Rationale: familiar tooling (Solution Explorer, NuGet UI, F5/Swagger debugger) reduces friction while learning new AI service patterns — migrate to VS Code once primitives are solid. Build targets numbered by part (1a–1f, 2a–2e, etc.) to enable plan-to-code traceability and future GitHub repo commit alignment. First milestone logged: 1a complete — Document Intelligence extraction working end-to-end, 200 OK confirmed in Swagger on Mar 10.
